Relational Database Service (RDS) Setup

Private MySQL instance — secure, isolated, EF Core migrated
AWS Console Step 1.1 — Create a VPC with Public & Private Subnets

Your database must live in a private subnet (no internet gateway route). Create the VPC first if you don't already have one with the right layout.

  1. Open VPC → Create VPC. Choose VPC and more.
  2. Name it mini-amazon-vpc. CIDR: 10.0.0.0/16.
  3. Set 2 Availability Zones, 2 public subnets (ALB & ECS), 2 private subnets (RDS).
  4. Enable a NAT Gateway in at least 1 AZ (needed so ECS tasks in private subnets can pull ECR images).
  5. Click Create VPC.
ⓘ
Note the IDs for later: VPC ID, both public subnet IDs (ALB), both private subnet IDs (RDS).
Security Step 1.2 — Create Security Groups

You need three security groups. Create them under VPC → Security Groups → Create security group.

NameInbound RulePurpose
sg-alb HTTP 80 from 0.0.0.0/0
HTTPS 443 from 0.0.0.0/0		 Public load balancer
sg-backend TCP 8080 from sg-alb .NET API containers
sg-rds MySQL/Aurora 3306 from sg-backend RDS — only backend can connect
sg-frontend TCP 3000 from sg-alb Next.js containers
⚠
Never add a rule that opens port 3306 to the internet (0.0.0.0/0). The RDS security group must only allow traffic from sg-backend.
AWS Console Step 1.3 — Create an RDS Subnet Group

RDS needs to know which subnets it can use. Navigate to RDS → Subnet groups → Create DB subnet group.

  1. Name: mini-amazon-rds-subnet-group
  2. VPC: select mini-amazon-vpc
  3. Add both private subnets (the ones without internet gateway routes)
  4. Click Create
AWS Console Step 1.4 — Provision the RDS MySQL Instance

Navigate to RDS → Create database and use these settings:

Terminal Step 1.5 — Apply EF Core Migrations to RDS

Because the RDS instance is in a private subnet you cannot connect to it from your laptop directly. Use one of these approaches:

✓
Recommended: Run migrations automatically at container startup inside ECS. Add the following to the backend's Program.cs before app.Run().
C# — Program.cs 
// Apply pending migrations on startup (runs inside ECS inside the VPC)
using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    db.Database.Migrate();
}

Alternatively, use an EC2 bastion host or AWS Systems Manager Session Manager to tunnel into the private subnet and run:

Shell 
# From within the VPC (EC2 bastion or SSM tunnel)
dotnet ef database update \
  --project ECommerce.Infrastructure \
  --startup-project ECommerce.API \
  --connection "Server=<RDS-ENDPOINT>;Port=3306;Database=miniamazon_db;User=admin;Password=<PWD>;"

Containerization & Elastic Container Registry (ECR)

Production Dockerfiles for Next.js & .NET 10, pushed to ECR
Code Step 2.1 — Backend Dockerfile (backend/Dockerfile)

Create this file at the root of the backend/ folder (one level above ECommerce.API/). It uses a multi-stage build targeting .NET 10.

Dockerfile 
# ── Stage 1: Build ──────────────────────────────────────────
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src

# Copy solution and restore (layer-cache friendly)
COPY ECommerce.Domain/ECommerce.Domain.csproj           ECommerce.Domain/
COPY ECommerce.Application/ECommerce.Application.csproj ECommerce.Application/
COPY ECommerce.Infrastructure/ECommerce.Infrastructure.csproj ECommerce.Infrastructure/
COPY ECommerce.Service/ECommerce.Service.csproj         ECommerce.Service/
COPY ECommerce.API/ECommerce.API.csproj                 ECommerce.API/
RUN dotnet restore ECommerce.API/ECommerce.API.csproj

# Copy all source and publish
COPY . .
RUN dotnet publish ECommerce.API/ECommerce.API.csproj \
      -c Release -o /app/publish --no-restore

# ── Stage 2: Runtime ────────────────────────────────────────
FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS runtime
WORKDIR /app
EXPOSE 8080

ENV ASPNETCORE_URLS=http://+:8080
ENV ASPNETCORE_ENVIRONMENT=Production

COPY --from=build /app/publish .
ENTRYPOINT ["dotnet", "ECommerce.API.dll"]
⚠
Do not copy appsettings.json with your local connection string into the image. The connection string will be injected at runtime via SSM in Phase 3.
Code Step 2.2 — Frontend Dockerfile (frontend/Dockerfile)

Next.js 16 supports a standalone output mode which drastically reduces the final image size. Enable it first in next.config.ts:

next.config.ts 
const nextConfig = {
  output: 'standalone',
};
export default nextConfig;

Then create frontend/Dockerfile:

Dockerfile 
# ── Stage 1: Dependencies ────────────────────────────────────
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json* ./
RUN npm ci --omit=dev

# ── Stage 2: Build ───────────────────────────────────────────
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .

# Injected at build-time so Next.js can embed API URL in the bundle
ARG NEXT_PUBLIC_API_URL
ENV NEXT_PUBLIC_API_URL=$NEXT_PUBLIC_API_URL

RUN npm run build

# ── Stage 3: Runtime ─────────────────────────────────────────
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production

COPY --from=builder /app/public            ./public
COPY --from=builder /app/.next/standalone  ./
COPY --from=builder /app/.next/static      ./.next/static

EXPOSE 3000
CMD ["node", "server.js"]

Also create a .dockerignore in the frontend/ folder:

.dockerignore 
node_modules
.next
.git
*.md
AWS CLI Step 2.3 — Create ECR Repositories

Run these commands (replace us-east-1 and your AWS account ID):

PowerShell / Bash 
# Set your region and account ID
$AWS_REGION  = "us-east-1"
$ACCOUNT_ID  = "$(aws sts get-caller-identity --query Account --output text)"

# Create repos
aws ecr create-repository --repository-name mini-amazon-backend  --region $AWS_REGION
aws ecr create-repository --repository-name mini-amazon-frontend --region $AWS_REGION
Terminal Step 2.4 — Build, Tag & Push Images to ECR
PowerShell / Bash 
# 1. Authenticate Docker to ECR
aws ecr get-login-password --region $AWS_REGION | `
  docker login --username AWS --password-stdin `
  "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com"

# 2. Build and push BACKEND
docker build -t mini-amazon-backend ./backend
docker tag  mini-amazon-backend:latest `
  "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/mini-amazon-backend:latest"
docker push "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/mini-amazon-backend:latest"

# 3. Build and push FRONTEND
#    ALB DNS is not known yet — use a placeholder or the future ALB DNS
docker build `
  --build-arg NEXT_PUBLIC_API_URL=http://<ALB-DNS>/api `
  -t mini-amazon-frontend ./frontend
docker tag  mini-amazon-frontend:latest `
  "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/mini-amazon-frontend:latest"
docker push "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/mini-amazon-frontend:latest"
⚠
You'll know the ALB DNS name only after Phase 4. You can use a placeholder value for now and rebuild the frontend image after Phase 4 with the real URL.
3

Container Orchestration via ECS (Fargate)

Serverless containers with secrets injected from SSM Parameter Store
Security Step 3.1 — Store the Connection String in SSM Parameter Store

Never put your database password in a Dockerfile or environment variable you hardcode. Store it securely in SSM and ECS will inject it at runtime.

AWS CLI 
# Store as SecureString (encrypted with KMS)
aws ssm put-parameter `
  --name "/mini-amazon/prod/DB_CONNECTION" `
  --type "SecureString" `
  --value "Server=<RDS-ENDPOINT>;Port=3306;Database=miniamazon_db;User=admin;Password=<PWD>;" `
  --region $AWS_REGION
IAM Step 3.2 — Create an ECS Task Execution IAM Role

ECS tasks need permission to pull images from ECR and read SSM parameters.

  1. Go to IAM → Roles → Create role
  2. Trusted entity: AWS service → Elastic Container Service Task
  3. Attach policies:
    • AmazonECSTaskExecutionRolePolicy (ECR pull, CloudWatch logs)
    • Create a custom inline policy to allow ssm:GetParameters on your parameter ARN
  4. Name it mini-amazon-ecs-task-exec-role
JSON — Inline SSM Policy 
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["ssm:GetParameters", "kms:Decrypt"],
    "Resource": [
      "arn:aws:ssm:us-east-1:<ACCOUNT_ID>:parameter/mini-amazon/prod/*"
    ]
  }]
}
AWS Console Step 3.3 — Create the ECS Cluster
  1. Go to ECS → Clusters → Create Cluster
  2. Name: mini-amazon-cluster
  3. Infrastructure: AWS Fargate (serverless)
  4. Click Create

Go to ECS → Task definitions → Create new task definition.

Under Environment variables, add the connection string from SSM:

JSON — Task Definition Secrets Section 
"secrets": [
  {
    "name":      "ConnectionStrings__DefaultConnection",
    "valueFrom": "arn:aws:ssm:us-east-1:<ACCOUNT_ID>:parameter/mini-amazon/prod/DB_CONNECTION"
  }
]
✓
ASP.NET Core reads environment variables named ConnectionStrings__DefaultConnection (double underscore = colon) automatically, so no code changes are needed.

Application Load Balancer & Routing

Single public entry point — route /* to frontend, /api/* to backend
AWS Console Step 4.1 — Create Two Target Groups

Go to EC2 → Target Groups → Create target group.

Target Group 1 — Backend:

SettingValue
Target typeIP (required for Fargate)
Nametg-backend
Protocol / PortHTTP / 8080
VPCmini-amazon-vpc
Health check path/api/health (or any endpoint that returns 200)

Target Group 2 — Frontend:

SettingValue
Target typeIP
Nametg-frontend
Protocol / PortHTTP / 3000
VPCmini-amazon-vpc
Health check path/
✓
Leave the targets list empty for now. ECS will register task IPs automatically when you attach the target groups to services in Step 3.6.
AWS Console Step 4.2 — Create the Application Load Balancer

Go to EC2 → Load Balancers → Create load balancer → Application Load Balancer.

SettingValue
Namemini-amazon-alb
SchemeInternet-facing
IP address typeIPv4
VPCmini-amazon-vpc
SubnetsBoth public subnets
Security groupsg-alb
ListenerHTTP : 80 (default action → tg-frontend)

After creation, note the ALB DNS name — this is your app's public URL.

AWS Console Step 4.3 — Add Routing Rule for /api/*

Open EC2 → Load Balancers → mini-amazon-alb → Listeners → HTTP:80 → Manage rules.

  1. Click Add rule
  2. Add condition: Path is /api/*
  3. Action: Forward to tg-backend
  4. Priority: 1 (higher priority than the default)
  5. Save

The final listener rule order:

Listener Rules (HTTP:80) 
Priority 1  │  IF path = /api/*   │  FORWARD → tg-backend
Default     │  All other traffic  │  FORWARD → tg-frontend
AWS Console Step 4.4 — Attach Target Groups to ECS Services

Go back to your ECS Services (created in Step 3.6) and edit each one to attach the load balancer:

  1. backend-service → Load balancer: mini-amazon-alb, Container: 8080, Target group: tg-backend
  2. frontend-service → Load balancer: mini-amazon-alb, Container: 3000, Target group: tg-frontend
✓
ECS will automatically register and deregister task IPs into the target groups as tasks start and stop.
Terminal Step 4.5 — Rebuild Frontend Image with Real ALB URL

Now that you know the ALB DNS name, rebuild and push the frontend image with the correct API URL:

PowerShell / Bash 
# Replace <ALB-DNS> with e.g. mini-amazon-alb-1234567890.us-east-1.elb.amazonaws.com
docker build `
  --build-arg NEXT_PUBLIC_API_URL=http://<ALB-DNS>/api `
  -t mini-amazon-frontend ./frontend

docker tag mini-amazon-frontend:latest `
  "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/mini-amazon-frontend:latest"

docker push "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/mini-amazon-frontend:latest"

# Force ECS to pull the new image
aws ecs update-service `
  --cluster mini-amazon-cluster `
  --service frontend-service `
  --force-new-deployment `
  --region $AWS_REGION
